# 관리자/클라이언트 도메인 격리 배포 테스트 가이드 (Deployment & Test Guide)

본 가이드는 일반 사용자(Client) 서비스 도메인과 관리자(Admin) 전용 서브도메인을 격리하는 설계 사양에 따른 로컬 테스트 환경 설정, 프로덕션 배포 설정, 그리고 보안 차단 시나리오 검증 방법에 대해 상세히 다룹니다.

---

## 1. 개요
* **목적**: 클라이언트 페이지와 관리자 페이지를 독립된 도메인으로 매핑하고, 일반 사용자 도메인으로의 관리자 경로 접근을 원천 봉쇄(404 응답)하여 보안을 강화합니다.
* **접근 격리 요약**:
  * `CLIENT_HOST` (예: `localhost:3000`) -> 일반 상품 구매, 장바구니, 주문 조회 기능만 접근 가능. `/admin` 경로 접근 시 **404 Not Found**로 차단 및 은폐.
  * `ADMIN_HOST` (예: `admin.localhost:3000`) -> 관리자용 대시보드, 주문 및 상품 관리 기능 접근 가능. 미들웨어의 토큰 검증 작동.

---

## 2. 로컬 개발 환경 테스트 가이드

### 2.1. 로컬 다중 도메인 매핑
현대의 크롬(Chrome), 사파리(Safari), 엣지(Edge) 등 주요 브라우저는 `*.localhost` 서브도메인을 별도 설정 없이 자동으로 로컬 루프백 주소(`127.0.0.1`)로 리졸브(Resolve)합니다. 

따라서 로컬에서도 별도의 `/etc/hosts` 설정 없이 **`http://localhost:3000`** 및 **`http://admin.localhost:3000`** 주소로 각각 다중 도메인 테스트가 즉시 가능합니다.

> [!NOTE]
> 만약 구형 브라우저 또는 환경 특성으로 인해 `admin.localhost` 접속이 정상적으로 이루어지지 않는 경우, 아래와 같이 로컬 호스트 설정을 추가해 주십시오.
> 
> * **macOS / Linux**:
>   `sudo nano /etc/hosts` 명령을 통해 아래 행을 추가하고 저장합니다.
>   ```text
>   127.0.0.1 admin.localhost
>   ```
> * **Windows**:
>   `C:\Windows\System32\drivers\etc\hosts` 파일을 메모장(관리자 권한)으로 열어 동일한 행을 입력하고 저장합니다.

### 2.2. 환경 변수 설정 (.env)
프로젝트 루트 폴더 내의 `.env` 파일에 격리에 사용할 호스트 정보를 정의합니다.
```env
# 로컬 개발 환경용 설정
CLIENT_HOST=localhost:3000
ADMIN_HOST=admin.localhost:3000
```

## 3. 프로덕션 배포 설정 가이드 (Vercel 예시)

프로덕션 환경으로 배포 시 단일 빌드 상태에서 다중 도메인을 바인딩하는 방법입니다.

### 3.1. DNS 및 Vercel 도메인 추가
1. **도메인 구매 및 네임서버 설정**: 일반 사용자 서비스용 루트 도메인(예: `shop.com`)을 등록합니다.
2. **Vercel Project Settings > Domains** 메뉴로 이동합니다.
3. 두 개의 도메인을 추가하고 동일한 프로젝트로 리다이렉트 없이 바인딩합니다:
   - `shop.com` (일반 사용자 도메인)
   - `admin.shop.com` (관리자 전용 서브도메인)
4. Vercel이 제공하는 CNAME 및 A 레코드 정보를 본인의 DNS 설정에 성실히 등록하여 도메인 연결을 활성화합니다.

### 3.2. Vercel 환경 변수 세팅
Vercel Project Settings > Environment Variables 메뉴에 프로덕션용 호스트 환경 변수를 설정합니다.
* `CLIENT_HOST` = `shop.com`
* `ADMIN_HOST` = `admin.shop.com`
* (주의) Vercel에서 각 환경 변수가 Production, Preview, Development 모든 환경에 맞게 동적으로 세팅되거나 로컬용 설정과 충돌하지 않도록 주의하여 입력합니다.

## 4. 보안 차단 검증 테스트 시나리오

배포 완료(또는 로컬 서버 구동) 후, 아래 시나리오를 통해 도메인 격리 및 보안 작동을 철저히 검증합니다.

### 4.1. [시나리오 1] 일반 사용자 도메인을 통한 무단 관리자 경로 진입 차단 (성공 조건: 404)
1. 웹 브라우저를 열고 일반 사용자 주소인 **`http://localhost:3000/admin`** 또는 **`http://localhost:3000/admin/orders`**로 접속합니다.
2. **검증 결과**: 페이지가 열리지 않고 브라우저에 **`404 Not Found`** 또는 Next.js 기본 404 에러 화면이 나타나야 합니다.
3. API 차단 검증: Postman 등 도구를 통해 `GET http://localhost:3000/api/admin/orders` API를 호출합니다.
4. **검증 결과**: 응답 코드가 `404 Not Found` 혹은 접근 불가능한 응답 상태가 반환되어야 합니다.

### 4.2. [시나리오 2] 관리자 도메인을 통한 정상 관리자 기능 진입 (성공 조건: 정상 인가)
1. 웹 브라우저를 열고 관리자 전용 주소인 **`http://admin.localhost:3000/admin`**으로 접속합니다.
2. **검증 결과**: 미들웨어가 해당 호스트를 허용된 `ADMIN_HOST`로 올바르게 인지하고, 로그인 세션 검증 프로세스로 정상 통과시킵니다. (세션 토큰이 없으므로 `/admin/login` 화면으로 리다이렉트되어야 합니다.)
3. 관리자 아이디/비밀번호로 로그인한 후 관리 대시보드 및 주문 내역이 올바르게 로드되는지 확인합니다.
